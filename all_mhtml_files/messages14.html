The main problem of that issue has been fixed in 3.0.0 version.
To sum up, how it works:

If you put just a hostname, 'https://' prefix will be added
Then desktop app adds / at the end of url, so time.acme.corp trasnforms to https://time.acme.corp/ at this point
Next, app tries to request status endpoint (https://time.acme.corp/status), but prior to 3.0.0 version it was /status => https://time.acme.corp//status
If nothing found - we try cattr.manifest file (more info below) => (https://time.acme.corp/cattr.manifest), but prior to 3.0.0 version it was /cattr.manifest => https://time.acme.corp//cattr.manifest
The last attempt is api/status => https://time.acme.corp/api/status
So, basically, desktop app searches for /api/status endpoint - it will mark the 'root' of API that it needs. You can check yourself is that endpoint available on your instance by yourself. There should be some JSON with a cat. =)
Some words about cattr.manifest file. It should be automatically generated when you build cattr fronted (with docker it is ships already builded). It is json file, that contain some basic information about cattr & frontend, but mainly - backend_path key, that can help desktop app to find backend 'root'.

Finally, you can:

Check /api/status endpoint by yourself (it seems, that prior 3.0 version it is only stable variant)
Make some rewrites in nginx config to properly work with cattr.manifest file at frontend, and check this file for correct path to your backend
Check your SSL certificates. Not, that if you don't have redirect http->https at your server config and you put full url to hostname filed - it should work with http without extra question
